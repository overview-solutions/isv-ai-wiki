Landscape & layers · utility AMI vs village minimum · Jun 2026

Summary on overview page Two goals, problem bullets, and nested page index live on Metering focus · Overview. This page is the detailed landscape — diagrams and layer tables.

Utility AMI and village mini-grids share the same ecosystem layers — payment schemes, semantic registers, exchange formats, protocols, and physical access — but utilities must implement every layer deeply while villages can narrow the contract at the gateway. The diagram below is OSI-inspired (not a literal OSI map). Terms: acronym list · VMRS tiers.

Metering landscape — layered ecosystem (utility vs village)

Read across: each row is a layer type. Amber = utility-heavy scope. Green = village minimum. Teal = both need it, different depth. Do not confuse layers — STS is a payment/key scheme (L6/L3), OBIS is register semantics (L5), DLMS is a meter protocol (L2), MQTT is northbound messaging (L2/L4).

Ecosystem actors (who owns which layer)

Customer→ CIU / SMS→ Meter (L1–L5)→ DCU / gateway (L2–L4)→ HES / platform (L4–L7)→ Operator / MDMS (L7)

Parallel paths: STSA/KMC (L3) issues STS keys to meters; mobile money API (L7) connects to vending without replacing meter reads; pole EMS (L2–L5) handles SunSpec DER separately from revenue VMRS.

How layers interact (not a flat list)

L6 vs L5 — payment scheme ≠ register format
STS (IEC 62055) defines token transfer and encryption; OBIS (IEC 62056-6-1) defines what “1.8.0 kWh” means. A meter can accept STS tokens while exposing credit on vendor-specific COSEM objects — see vendor ICD for tier B mapping.
L4 vs L5 — Profile Generic vs simple totals
Utilities use Profile Generic (COSEM class 7) to buffer 15-minute intervals for VEE. Villages can read totalizer registers (L5) on schedule without maintaining class-7 buffers — unless theft detection needs interval data.
L3 — association suites vs village gateway
Utility deployments use DLMS association + HLS per meter. A village DCU can hold one local DLMS association and publish only TLS MQTT northbound — shrinking the trust surface on expensive WAN links.
L3 — SIP (Session Information Protocol)
Glenn Algie’s draft layer for authenticated market sessions (DER trading, disconnect authorization) — sits above MQTT at L3, not to be confused with VoIP SIP or SunSpec.
Parallel DER track
Revenue meter VMRS (L5) and inverter SunSpec models (L5 DER) converge at the pole EMS (L2–L4) but should not be merged into one procurement column — see VMRS page exclusions.

What breaks in the field today

  • Cadillac procurement on Tata budgets — RFPs copy utility DLMS/MDMS language; vendors quote full AMI when the operator only needs prepaid + aggregate feeder visibility.
  • OEM key bottlenecks — proprietary RF stacks block second-source DCUs (see May 2026 Tech Comm notes).
  • Protocol translation tax — STS, DLMS, Modbus, and SunSpec coexist on one pole; no shared northbound schema until VMRS + MQTT profile.
  • DER vs revenue meter confusion — SunSpec on inverters solves generation monitoring; it does not replace STS vending or OBIS billing registers.
