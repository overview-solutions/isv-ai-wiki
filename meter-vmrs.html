Read this before citing VMRS in an RFP VMRS is an ISV Tech Comm working proposal — a shortlist of registers we think village operators need. It is not an IEC, DLMS UA, or STS published profile. See tier definitions and acronyms (including ICD). Every tier B mapping must be validated against the vendor ICD before procurement.

The primary artifact is the northbound JSON/MQTT profile — a gateway export contract. VMRS is the appendix: a checklist of likely OBIS keys in that payload. Most tier A codes already exist on every DLMS meter; the village gap is procurement scope and northbound shape, not new register inventions. See layered landscape and what the tiers mean.

What is an ICD?

When we say “validate against the vendor ICD,” we mean their Interface Class Description: the machine-readable description of which COSEM objects exist on a specific meter type, each with an OBIS logical name, class, and attributes.

Two meters can both be “DLMS/STS prepay” but expose different logical names for credit balance or disconnect state. Tier A OBIS codes like 1.8.0 are universal; tier B items (credit, tamper bits, relay) are where you open the ICD and ask: which object, which attribute, which read method?

Utility AMI teams already demand ICDs for certification. Village POCs need a short ICD excerpt — only the rows that map to VMRS targets — not the full hundred-page object list.

What the tiers mean — and what VMRS is for

VMRS is not a new OBIS standard. The tiers say how much each line item is “already industry fact” vs “ask the vendor ICD.”

Tier A · Standard OBIS (IEC 62056-6-1)

Examples: 1.8.0, 1.7.0, 32.7.0, 0.0.0. Universal metrology and clock codes — every DLMS vendor already implements these. ISV does not introduce alignment here; we cite them as payload keys in the northbound profile.

Value: Procurement checklist and JSON schema labels — “we need these reads,” not “we invented these codes.”

Tier B · Common COSEM practice

Examples: remaining credit, disconnect control, device serial, tamper status. Appear on most STS/DLMS prepay meters, but logical names and bit meanings vary by ICD. VMRS names the semantic target; only a vendor ICD sheet harmonizes the path to it.

Value: Anchor-led OEM ask — “document ICD mapping for meters you already ship to us.” The gap matrix per vendor is the real deliverable.

Tier C · ISV working proposal

Example: last token / transfer audit register. Operator need may be real; OBIS assignment is not standardized. Optional — drop from POC unless an anchor operator requires STS audit northbound.

Value: Low. Do not build a strategy around tier C alone.

So is VMRS a useful exercise? Yes, if reframed: align on northbound JSON/MQTT + scope-limited procurement (“not full MDMS/Profile Generic”). The OBIS table is mostly tier A bibliography plus tier B ICD homework. POC success = gateway publishes northbound-mqtt-v0.1 on a real feeder; tier A confirmed; tier B gaps published — not “VMRS compliance.”

How the register list was chosen

  1. IEC 62056-6-1 (OBIS) — Universal metrology and clock codes (tier A): e.g. 1.8.0 active import energy is the de-facto worldwide billing register; not an ISV invention.
  2. DLMS UA COSEM / prepay meter practice — Tier B codes (credit balance, disconnect control, tamper status) appear on STS-capable DLMS meters but logical names vary by manufacturer ICD — we document common patterns, not one true mapping.
  3. IEC 62055 (STS) — Defines token transfer and encryption. STS is not OBIS-native; mapping “remaining credit” to an OBIS code is vendor-specific (tier B).
  4. ISV Tech Comm field requirements — Theft visibility, prepaid parity, relay state, asset identity from May 2026 notes and operator interviews — tier C where no single OBIS is standardized.
  5. Vendor study citations — Inhemeter brochure cites OBIS/IEC62056-21 optical read (meter-vendor-study [s15]) — supports that DLMS meters expose standard OBIS, not that VMRS as a bundle is vendor-certified.

What we did not do: hands-on ICD extraction on target meters, DLMS UA conformance testing, or STSA register harmonization. Those are the next validation steps.

Explicitly excluded from VMRS

  • Full Profile Generic (class 7) interval buffers — utility L4; add only if theft analytics need it
  • Full COSEM ICD — utility certification scope
  • STS key management (STSA/KMC) — separate L3 trust domain
  • SunSpec inverter registers — parallel DER track on pole EMS

Northbound profile v0.1 — where it lives

The JSON/MQTT profile lives in the repo under meter-benchmark/ alongside VMRS registers — not in a database schema or buried in vendor PDFs. Rationale:

  • Versioned with the wiki (Git history, PR review, GitHub Pages deploy).
  • Edge firmware and platform teams can fetch stable URLs from Pages.
  • Human docs (Problems today, this page) link to the same files agents and CI can consume.

MQTT topic root: openami/v1/{feederId}/{gatewayId}/meter/{meterSerial}/telemetry — see full topic tree in the JSON file.

SIP — Session Information Protocol (Glenn Algie)

Not SunSpec SIP · not VoIP SIP Glenn’s Session Information Protocol is an ISV Tech Comm proposal for establishing authenticated sessions between meters, gateways, and market participants in decentralized DER energy markets — so disconnect commands and settlements are bound to a session, reducing spoofing and theft in peer-to-peer trading.

Northbound profile v0.1 reserves:

  • Optional sessionId on telemetry payloads (traceability).
  • Mandatory session handshake on command topics before relay/disconnect or settlement actions.
  • Topic: {topicRoot}/session/{sessionId} with participants, feeder scope, validity window, signature.

Normative SIP specification is evolving; this wiki tracks hooks in JSON until Glenn publishes SIP v0.1 text.

Unifying manufacturers — logical approach

  1. Lead with northbound MQTT/JSON, not “adopt VMRS.” Ask: “Can you emit northbound-mqtt-v0.1 from your DCU or document an ICD excerpt for these reads?”
  2. Separate revenue meter track from DER track. VMRS + MQTT for billing; SunSpec/MQTT on pole EMS for generation (see ISV-HCK-001 / Stonehenge topology).
  3. Edge normalization contract. Vendor keeps proprietary NAN; gateway publishes northbound envelope with tier A keys and documented tier B mappings.
  4. 90-day POC. One feeder, 10–30 meters, northbound export on schedule, STS vending unchanged, gap matrix published.
  5. Publish gaps in vendor study. Upgrade VMRS column from — to doc only with ICD citation.

Technical POCs to find in each company

Skip generic sales@ — ask for these roles by title or department:

Firmware / metrology lead Owns OBIS register implementation, tamper flags, relay control. Can confirm 96.5.0 bit meanings and read latency.
DLMS / AMI protocol engineer Association objects, Profile Generic, which COSEM classes expose VMRS. Often in “smart meter R&D” or “communication module” team.
STS / prepay product manager Token gateway, 96.8.0 mapping, CIU integration, STSA key process — critical for SSA village deployments.
DCU / gateway software lead Store-and-forward, MQTT/REST northbound, multi-vendor meter drivers. SteamaCo/Nimbus, Donsun DSC100, pole EMS teams.
Regional integrator / utility AMI PM Africa or South Asia field references, installed-base OBIS tables, partner portal access (Inhemeter-style gated docs).
DER / EMS partner engineer For Glenn’s SIP and DER markets — often separate from revenue meter team; may sit with inverter or microgrid platform group.
